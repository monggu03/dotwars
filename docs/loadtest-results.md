## 한계점 분석

### 1. 어디서 처음 깨졌는가
**안 깨졌다.** 1500 VU(364 req/s)까지 에러 0%, p95 23ms, max 80ms.
CPU 는 최대 46% 에 그쳤고 메모리·커넥션 모두 여유. 즉 **이 워크로드에서 한계점에 도달하지 못함.**

### 2. 왜 안 깨졌는가 (병목 미도달 분석)
- **CPU**: 2 vCPU 중 피크 46% (≈ 1 코어). c7i-flex 스로틀 근처도 안 감.
- **메모리**: Java RSS 467→520MB, 단 53MB 증가. swap 0. 힙 여유 충분.
- **TCP**: 1500 동접 ESTABLISHED 정상 수용 (Tomcat 기본 스레드 200 + NIO 라 동접≠스레드, 커넥션 큐로 처리).
- **읽기 경로가 가벼움**: canvas/stats 는 Redis 조회 + 1초 캐시. DB 거의 안 침. → CPU 거의 안 씀.
- **클라이언트 한계도 아직 안 옴**: k6 가 1500 VU 를 정상 생성(에러 0), 노트북/네트워크도 364 req/s·95MB 수신 무난.

### 3. 개선/추가 검증 카드
- 진짜 한계를 보려면: ① VU 를 3000~5000 로↑ (이땐 노트북·집인터넷이 먼저 한계일 가능성) ② **쓰기 경로(페인트) 부하** — JWT 인증 + ACTIVE 세션 + DB INSERT 라 읽기보다 무거움. 실제 병목은 여기일 확률이 높음.
- 튜닝 카드(필요 시): Tomcat `threads.max`, HikariCP pool, Redis Lettuce pool, canvas 응답 캐시, c7i-flex→c7i 고정성능, Nginx keepalive/worker_connections.

### 4. 실제 운영 트래픽 대비 안전 마진
- 본 축제 예상 100명 ≈ 60 req/s.
- 검증된 안정 처리량: **최소 364 req/s @ CPU 46%** (한계 아님, 그 이상도 가능).
- 안전 마진: **최소 6배 이상** (실제론 훨씬 더 — CPU 가 절반밖에 안 찼으므로).

### (옛 체크리스트 — 이번엔 해당 없음)
- **Tomcat 스레드 풀**: `server.tomcat.threads.max` (기본 200) — 동접 많으면 상향
- **HikariCP(DB 풀)**: `spring.datasource.hikari.maximum-pool-size` — 읽기 위주면 큰 영향 적음
- **Redis 커넥션(Lettuce)**: 기본 공유 커넥션 — 부족 시 풀 설정
- **stats 캐시**: 이미 1초 TTL 캐시 있음 → canvas 도 짧은 캐시 검토
- **인스턴스**: c7i-flex → c7i(고정 성능) 또는 상위 vCPU 로 스케일업
- **앞단**: Nginx `worker_connections`, keepalive 튜닝 / CDN 캐싱(정적 canvas)

### 4. 실제 운영 트래픽 대비 안전 마진
- 본 축제 예상: 100명. 100명 = 약 60 req/s (이전 측정).
- 한계 관측치: (채울 것) req/s 까지 안정.
- 안전 마진: 약 ( )배.

>  N VU(약 M req/s)까지 p95 Xms·에러 0% 로 안정적이었고, K VU 구간에서 (CPU 100% / 클라 병목 /
>  커넥션 상한) 때문에 p95 가 급증했다. 핵심은 단순히 '깨졌다'가 아니라, EC2 CPU 모니터링을 병행해
>  '서버 한계'와 '부하 생성기(노트북) 한계'를 구분한 점이다. 실제 운영 규모(100명, 60 req/s)
>  대비 약 ()배의 마진이 확인됐고, 추가 확장이 필요하면 Tomcat 스레드 풀 상향 또는 인스턴스
